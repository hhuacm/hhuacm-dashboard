# 项目协作准则

这份文档面向参与本项目的 AI / LLM 协作者，也面向后来阅读代码的人。它聚焦本项目期待的判断方式和协作气质，具体实现细节由当前代码、`DESIGN.md` 和真实需求共同约束。

## 项目取向

HHUACM Dashboard 长期处于测试和演进状态，因此更需要保持结构干净、表达准确和阅读顺畅。越是在快速变化中，代码、文档和界面越需要让后来者容易继续理解和修改。

一次好的改动应当让人能够沿着真实业务流程自然读懂：相关概念为什么存在，数据从哪里来，状态如何变化，失败时如何呈现，用户下一步可以做什么。项目重视这种阅读质感，也重视实现背后的秩序感。命名、文件边界、函数职责、数据流和界面层级都应当帮助读者建立稳定的系统模型。

本项目追求的精致感来自清楚的结构、可靠的状态、准确的文案和一致的组件使用。代码和界面都应当克制而完整，复杂性只在真实问题需要时出现，并被安放在读者容易找到、容易验证的位置。

## 协作方式

用户的表达可能是探索性的，也可能使用了尚未完全准确的工程术语。LLM 应结合项目背景、当前代码、用户的真实目标和可能的误解来理解需求。面对会影响架构、数据流、用户体验或长期维护方式的改动，应先说明自己理解到的目标、准备采用的实现方向和主要取舍，再开始执行。小范围、低风险、意图明确的修改可以直接完成，并在结果中交代关键变化。

当用户提出的具体实现与项目取向冲突时，应明确指出冲突，并给出更符合项目气质的推荐方案。协作中的解释应帮助用户获得更准确的判断，同时保持任务向前推进。需要确认时，问题应当短而具体；已经可以从代码和上下文判断的内容，应主动推进。

AI 辅助开发是本项目的重要协作方式。代码结构、文档表达和组件边界都应当让后续 LLM 能够稳定延续当前模式。重要取舍应体现在命名、文件边界、文档或代码结构中，避免停留在只有本轮对话才知道的隐式约定里。

## 架构与代码

本项目采用轻量级 DDD 与准函数式设计倾向。这里的「准函数式」指向清楚、可推理、低副作用的数据变换：输入、输出、依赖和副作用都应当容易看见；数据库、网络请求、时间、环境变量、React state 和 mutation 等副作用应集中在清楚的边界里。

项目的包边界受整洁架构和六边形架构启发。`apps/*` 应主要作为进程入口、运行时组装和外部适配器；`packages/api` 承担 tRPC/HTTP 适配；`packages/application` 承载应用用例、系统任务、后台刷新和外部 OJ 同步；`packages/domain` 保存纯业务事实；`packages/db`、`auth`、`env` 等包提供基础设施能力。依赖方向应尽量让业务用例远离具体 transport，避免 application 反向依赖 API、server、web 或 worker 入口。

稳定的业务规则应进入清晰的领域边界，临时或页面局部的规则应留在靠近使用处的位置。抽象可以在模式已经稳定、概念已经明确时出现；一个好的抽象应当让调用处更像业务伪代码，把泛型、适配层和技术细节收束在读者不必频繁触碰的位置。

代码应优先表达业务流程。TypeScript 的类型用于说明边界和数据形状，局部实现可以依赖推导减少噪音。没有系统学习过 TypeScript 的读者，也应当能像读带约束的伪代码一样理解主要流程。复杂类型、过深泛型和技巧化写法只有在它们明显降低整体复杂度时才值得引入。

函数和组件的拆分服务于阅读。helper 只有在名称能表达真实业务含义、能降低阅读负担或承载稳定规则时才出现。简单 React 组件保持内聚；复杂组件可以把稳定、纯粹、可命名的业务计算外提。UI 状态和交互流程高度耦合时，可以留在组件内部，让读者沿着页面行为直接理解。

内容需要由当前真实需求支撑。配置项、兼容层、迁移兜底、适配层、重试策略和未来扩展接口都应来自已经出现的真实复杂性。

## 前端与界面

前端界面遵循工具型产品的取向：用户应能快速判断当前位置、理解信息状态并完成操作。公开入口可以有更强的识别度，后台页面应保持稳定、克制和信息优先。具体视觉、布局、颜色、组件和交互准则以 `DESIGN.md` 为准。

界面实现应优先使用项目已有组件和 HeroUI V3 组件模式。页面应呈现真实状态，包括加载、错误、空状态、保存中、删除确认等用户会实际遇到的情况。视觉上的精致感应服务于信息关系和操作效率。

## 文档

文档应说明真实取舍、项目背景和协作方式，帮助读者建立系统模型。说明可以使用连续、自然的段落；命令、路径、边界和关键规则需要准确清楚。

注释只在复杂逻辑确实需要帮助时出现。代码本身应承担主要解释责任，文档补充代码无法自然表达的背景、约束和取舍。

## 常用命令

本项目使用 Ultracite 做格式化和静态检查。完成代码或文档修改后，默认运行完整验证：

```bash
bun install
bun run dev
bun run verify
```

`verify` 会先运行自动修复，再执行类型检查、风格检查和全部测试。如果自动修复改动了文件，应检查 diff，确认这些变化属于当前任务。

数据库相关命令：

```bash
bun run db:local
bun run db:sync
```

提交前至少运行：

```bash
bun run verify
```

如果正在定位问题，可以按需拆开运行更细的底层命令；最终交付前以完整验证为准。
